# Contributing to @nano-step/eval-harness

Thanks for considering a contribution. This doc tells you exactly what helps, what doesn't, and how to land a PR fast.

## TL;DR for contributors

1. Pick a `good-first-issue` from [issues](https://github.com/nano-step/eval-harness/issues?q=is%3Aopen+is%3Aissue+label%3A%22good+first+issue%22)
2. Fork → branch → make change → add or update a test under `scripts/eval/tests/`
3. Run `for t in scripts/eval/tests/*.sh; do bash "$t"; done` — all 19+ must pass
4. Open a PR with a one-line summary + before/after evidence

## What we want

| Want | Don't want |
|---|---|
| Bug fixes from [`KNOWN_ISSUES.md`](./KNOWN_ISSUES.md) HIGH/MEDIUM list | Refactors with no observable behavior change |
| New check kinds (with a 3-case demo skill) | Features without a regression test |
| Portability fixes (BSD/macOS, alpine/musl, alternate `jq`/`yq` versions) | Bumping dependencies "just because" |
| Faster paths in the critical loop (lock, manifest, score) | Adding new runtime deps (Node, Python pkgs) — bash + jq + python3 stdlib only |
| Documentation that maps invariants to specific file:line | Documentation generated by an LLM with no examples |
| Audit reports that find new bugs (with reproducers) | Drive-by lint nits |

## Project shape

- **No daemon, no Node CLI, no Unix socket.** Bash + `jq` + `python3` stdlib only.
- **Honest failure modes.** When the harness can't judge, it says `verdict: null` and tells you why. Never fabricate.
- **`set -euo pipefail` everywhere.** New scripts must follow.
- **Every fix needs a test.** The 8 v0.4.2 BLOCKER fixes each landed with a regression test. Yours will too.
- **YAML schema changes are additive.** Removing or renaming a field is a major-version concern.

## Running the tests

```bash
# All suites (each one self-contained, fast):
for t in scripts/eval/tests/*.sh; do echo -n "$(basename $t .sh): "; bash "$t" 2>&1 | tail -1; done

# Single suite under inspection:
bash scripts/eval/tests/regression_inject.sh

# Smoke test the demo (no API spend):
EVAL_SKIP_AUTH_CHECK=1 eval-harness run --skill=omo-session-distiller --dry-run
```

## Writing a new test

Tests are bash scripts under `scripts/eval/tests/`. Each one is self-contained:

```bash
#!/usr/bin/env bash
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
REPO_ROOT="$(cd "$SCRIPT_DIR/../../.." && pwd)"
WORK="$(mktemp -d -t eval-harness-XXX.XXXXXX)"
trap 'rm -rf "$WORK"' EXIT

# ... arrange ephemeral state ...
# ... act (invoke the harness) ...
# ... assert ...

echo "PASS: <what was proven>"
exit 0
```

Patterns to follow:

- **Stub `opencode`** when you don't need a real LLM call (see `regression_inject.sh` for the canonical stub)
- **Stub `curl`** when testing `llm_judge` (see `llm_judge_unit.sh`)
- **Set `EVAL_SKIP_AUTH_CHECK=1`** to bypass the API-key preflight in tests
- **Set `EVAL_STATE_DIR="$WORK/state"`** to keep state isolated per test
- **Set `OPENCODE_SKILLS_ROOT="$WORK/skills"`** to keep skill discovery isolated

## Coding style

- **Shell**: bash, `set -euo pipefail`, prefer `[[ ]]` over `[ ]`, quote everything, use `local` for function vars
- **No `eval`, no `bash -c "$user_input"`** without going through `score_shell_is_unsafe` first
- **Comments only where load-bearing**: complex algorithms, security boundaries, portability quirks. Self-documenting code over docstrings.
- **One file = one concern.** New libs go in `scripts/eval/lib/`
- **Cross-platform.** Code must work on Linux + macOS (BSD coreutils). When in doubt, test both.

## Adding a new check kind

If you want to add e.g. `kind: http_response_contains`:

1. Add a `score_http_response_contains` function in `lib/score.sh` following the existing pattern
2. Add a `case` arm in `run_check`
3. Update `autofix.sh:propose_fix` to handle (or explicitly skip) the new kind
4. Add at least one positive and one negative test case
5. Document the new kind in README Layer 1 table + CHANGELOG

## How releases work

- `main` is always green (every test in `scripts/eval/tests/` passes)
- Features land on `feature/<vX.Y.Z>-<topic>` branches
- Releases happen via merge commit + annotated tag `vX.Y.Z` + GitHub Release
- semver: BLOCKER fixes are patch, new check kinds are minor, schema changes are major

## Reporting a bug

Open a [new issue](https://github.com/nano-step/eval-harness/issues/new) with:

1. **One-line summary** — what's broken
2. **Reproducer** — minimal case YAML / shell commands / env
3. **Expected vs actual** — what should happen, what does
4. **Environment** — `eval-harness --help | head -1`, `bash --version | head -1`, OS

Bonus points if you can point to the file:line where you think it's broken.
